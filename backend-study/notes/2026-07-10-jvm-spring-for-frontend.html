01

JVM — Java 세계의 런타임

한 줄로: JVM(Java Virtual Machine)은 Java 세계의 "런타임"이다. 브라우저나 Node가 JS를 실행해주듯, JVM이 Java·Kotlin 코드를 실행해준다.

"한 번 컴파일, 어디서나 실행"

TypeScript를 브라우저가 바로 못 읽어 JS로 변환하듯, Java/Kotlin도 중간 단계를 거친다. 결정적 차이는 그 중간 결과물(바이트코드)이 OS에 독립적이라는 점이다.

  1. 개발자가 .kt / .java 소스를 작성한다.
  2. 컴파일러가 바이트코드(.class·.jar)로 변환한다 — 사람이 읽는 코드도, 특정 CPU 기계어도 아닌 중간 언어.
  3. 각 OS의 JVM이 이 바이트코드를 받아, 그 CPU에 맞는 기계어로 실시간 변환(JIT)해 실행한다.

맥에서 만든 .jar를 리눅스 서버에 그대로 던져도, 각자 자기 OS용 JVM만 있으면 똑같이 돈다. JVM이 OS 차이를 흡수하는 추상화 레이어이기 때문이다. 이게 Java의 유명한 슬로건 "Write Once, Run Anywhere"다. 개발자가 브라우저별 JS를 따로 안 짜도 각 JS 엔진이 알아서 돌려주는 것과 같은 구조.

놓치기 쉬운 포인트 "JVM 언어"는 Java만이 아니다. Kotlin·Scala·Groovy·Clojure도 전부 바이트코드로 컴파일돼 JVM 위에서 돈다. TS·CoffeeScript가 결국 JS로 변환돼 같은 엔진에서 도는 것과 판박이. 백엔드가 "Kotlin으로 짰다"고 해도 실행 기반은 결국 JVM이다.

Node의 V8이 GC와 메모리 관리를 대신 해주듯, JVM도 메모리를 자동 관리(GC)한다. 직접 할당·해제하지 않는 감각은 JS와 완전히 같다. 협업 중 백엔드가 "힙 메모리", "GC 튜닝", "OOM" 얘기를 하면 — Node 서버가 메모리 누수로 죽는 것과 같은 층위의 문제라고 이해하면 된다.

02

Spring & Spring Boot

Spring은 React,
Spring Boot는 Next.js다.

이 비유가 놀랄 만큼 정확하다. 하나는 강력한 핵심 프레임워크, 하나는 그 위에서 설정을 다 잡아주는 상위 프레임워크.

Spring은 왜 쓰나

Spring은 Java 백엔드의 표준 프레임워크다. 핵심 기능은 DI(의존성 주입) — 객체 생성과 연결을 컨테이너가 대신 해준다(자세히는 03에서). 덕분에:

· 결합도가 낮아짐 — 부품 교체가 쉬움(테스트 시 가짜 DB로 교체 등)
· 보일러플레이트 제거 — 웹 서버·트랜잭션·보안·DB 연결 같은 "매번 똑같이 짜야 하는 것"을 생태계(Spring MVC·Data·Security)가 제공

즉 Spring의 존재 이유는 반복되는 인프라 코드를 표준화해 비즈니스 로직에만 집중하게 하는 것이다.

그럼 Spring Boot는 왜 또?

Spring은 강력하지만 설정이 지옥이었다. XML 수백 줄, 라이브러리 버전 궁합, 톰캣 따로 설치… 시작하기가 너무 무거웠다. 프론트에서 webpack·babel·라우터·SSR을 손으로 잡던 고통을 Next.js가 "설정 없이 바로 시작"으로 해결한 것 — 딱 그 관계다.

문제Spring Boot의 해결
설정이 방대함자동 설정 — 관례에 따라 알아서 세팅
라이브러리 버전 충돌Starter — 검증된 버전 세트가 한 번에 딸려옴
웹서버 따로 설치·배포톰캣 내장 — .jar 실행하면 바로 서버가 뜸
상태 확인·운영Actuator 등 운영 편의 기본 제공(04 참고)

요즘 신규 Java/Kotlin 백엔드는 사실상 100% Spring Boot로 시작한다. Spring만 쓰는 경우는 거의 없다. 관계식으로: Spring ↔ Spring Boot = React ↔ Next.js = 엔진 ↔ 완성차.

03

DI와 어노테이션

이 둘은 한 몸이다. 어노테이션은 "DI를 하라고 Spring에 알려주는 표시"다. DI를 먼저 잡고, 어노테이션이 그걸 어떻게 표현하는지 보자.

문제부터 — 객체가 필요한 걸 직접 new 한다

안티패턴 
class PostService {
    private EmailClient email = new EmailClient();               // 직접 생성
    private PostRepository posts = new PostRepository(new DbConnection("prod-db"));
}
class PostService {
    private val email = EmailClient()                     // new 없음, val
    private val posts = PostRepository(DbConnection("prod-db"))
}
class PostService {
  private email = new EmailClient();                  // TS도 직접 생성은 Java와 거의 동일
  private posts = new PostRepository(new DbConnection("prod-db"));
}

↑ 우측 상단 Java · Kotlin · TS 탭으로 언어 전환 (전체 코드블록 동기화 · TS는 NestJS 기준)

무엇이 문제인가 — 강하게 묶인다. 이 클래스가 "알림은 EmailClient로, DB는 prod-db로"까지 다 안다. 테스트할 때 가짜 이메일 클라이언트로 못 바꾸고, prod-db에 진짜로 붙어버린다. 게다가 EmailClient가 필요한 클래스가 50개면 50곳에서 각자 new — 만드는 방식이 바뀌면 50곳을 다 고쳐야 한다. 프론트로 치면 모든 컴포넌트가 fetch 클라이언트를 각자 new로 만드는 것과 같다.

해결 — 밖에서 주입받는다

생성자 주입 
class PostService {
    private final EmailClient email;
    private final PostRepository posts;

    // 내가 만들지 않고, 누군가 "주입"해준다
    PostService(EmailClient email, PostRepository posts) {
        this.email = email;
        this.posts = posts;
    }
}
// 주 생성자 파라미터에 val을 붙이면 필드 선언·대입이 한 번에 끝난다
class PostService(
    private val email: EmailClient,
    private val posts: PostRepository,
)
@Injectable()
class PostService {
  // constructor 파라미터의 private readonly = 선언·대입 한 번에 (Kotlin val과 동일)
  constructor(
    private readonly email: EmailClient,
    private readonly posts: PostRepository,
  ) {}
}

이제 이 클래스는 "이메일 발송 기능과 글 저장소가 필요하다"고 선언만 할 뿐, 그게 진짜인지 가짜인지·어떻게 만들어졌는지 모른다. 그럼 이 부품들을 누가 만들어 넣어주나? → Spring이다.

IoC 컨테이너 — Spring이 대신 조립한다

Spring은 앱이 뜰 때 필요한 객체(빈, Bean)들을 전부 미리 만들어 큰 상자에 담아둔다. 이 상자가 컨테이너, "생성·연결의 주도권을 개발자가 아니라 프레임워크가 갖는다"는 뜻에서 IoC(제어의 역전)라 부른다.

  1. "관리 대상이야"라고 표시된 클래스들을 스캔한다.
  2. 각각을 생성해 컨테이너에 보관한다.
  3. 생성할 때 생성자가 요구하는 부품을 컨테이너에서 찾아 자동으로 꽂아준다(주입).
프론트 비유 React Context Provider가 위에서 값을 만들어두면 하위 컴포넌트가 useContext로 받아 쓰는 것. "누가 만드는지"와 "쓰는 곳"이 분리돼 있다. Spring 컨테이너가 거대한 Provider인 셈이다. 개발자는 new PostService(...)를 한 번도 직접 쓰지 않는다.

여기서 @어노테이션이 등장

Spring은 "무엇을 관리할지", "무엇을 주입할지"를 어떻게 아나? → 어노테이션으로 알려준다. 어노테이션은 코드에 붙이는 메타데이터 라벨이다. 그 자체로 동작하는 게 아니라 "이걸 이렇게 취급해줘"라고 프레임워크에 지시하는 스티커 — 프론트의 데코레이터나 "use client" 같은 지시자에 가깝다.

스테레오타입 
@Service        // "얜 비즈니스 로직 담당 빈. 컨테이너에 등록해"
class PostService { ... }

@Repository     // "얜 DB 접근 담당 빈"
class PostRepository { ... }

@RestController // "얜 HTTP 요청 받는 빈"
class PostController { ... }
@Service        // 어노테이션은 Java와 100% 동일 (철자·위치·import)
class PostService { ... }

@Repository
class PostRepository { ... }

@RestController
class PostController { ... }
@Injectable()          // Spring @Service·@Repository ≈ NestJS @Injectable (구분 없음)
class PostService {}

@Injectable()
class PostRepository {}

@Controller('posts')   // ≈ @RestController + 기본 경로
class PostController {}

@Service·@Repository·@Controller는 기능적으론 거의 같다(다 "빈 등록"). 이름을 나누는 이유는 역할을 사람이 읽어서 알게 하려는 것 — 계층(웹/로직/데이터)이 코드에서 한눈에 보인다.

실제로 조립되는 모습

조립 + 라우팅 
@RestController
class PostController {
    private final PostService service;

    // 생성자가 PostService를 요구 → Spring이 컨테이너에서 찾아 자동 주입
    PostController(PostService service) {
        this.service = service;
    }

    @GetMapping("/posts/{id}")               // GET 요청을 이 메서드에 연결
    Post get(@PathVariable Long id) {          // URL의 {id}를 인자로 바인딩
        return service.findById(id);
    }
}
@RestController
class PostController(private val service: PostService) {   // 주 생성자로 주입

    @GetMapping("/posts/{id}")                       // GET 요청을 이 메서드에 연결
    fun get(@PathVariable id: Long): Post =          // 식(=) 본문, 반환 타입은 뒤에
        service.findById(id)
}
@Controller('posts')
class PostController {
  constructor(private readonly service: PostService) {}   // 생성자 주입 (개념 동일)

  @Get(':id')                              // ≈ @GetMapping("/posts/{id}")
  get(@Param('id') id: number): Post {   // ≈ @PathVariable
    return this.service.findById(id);
  }
}

벌어지는 일: @RestController가 클래스를 등록하고 리턴값을 JSON 응답으로 만든다 · 생성자의 PostService는 Spring이 미리 만들어둔 걸 자동 주입(개발자가 new 안 함) · @GetMapping은 프론트 라우터의 <Route path>처럼 URL을 메서드에 연결 · @PathVariable은 경로 값을 인자로 바인딩. 어노테이션 몇 개로 "라우팅 + 객체 조립 + JSON 직렬화"가 전부 선언적으로 처리된다.

코드 읽을 때 옛 코드에선 @Autowired를 필드에 붙여 주입하는 걸 자주 본다. "여기 주입해"라는 명시적 표시. 다만 요즘 권장은 위의 생성자 주입이고(생성자가 하나면 @Autowired 생략 가능), 필드 주입은 테스트가 어렵고 final을 못 써서 지양된다.
개념한 줄 정의프론트 대응
DI부품을 직접 만들지 않고 밖에서 주입받기props·context로 주입
IoC 컨테이너객체를 대신 만들고 연결해주는 상자거대한 Context Provider
어노테이션"이렇게 취급해줘" 메타데이터 라벨데코레이터 · "use client"
@Service 등"이 클래스를 빈으로 등록해"Context에 등록되는 값
@GetMapping"이 URL을 이 함수에 연결해"라우터의 path 매핑

왜 이렇게까지 하나 — 한 문장으로: 부품을 갈아끼우기 쉽게(테스트·유지보수), 조립·인프라 코드는 프레임워크에 맡기고 로직에만 집중하려고.

Java · Kotlin · TS — 같은 개념, 다른 문법

위 예시는 코드블록 우측 상단 Java / Kotlin / TS 탭으로 바꿔 볼 수 있다(한 번 고르면 전체가 그 언어로 전환된다). Java·Kotlin은 같은 JVM · 같은 Spring이라 개념·동작이 완전히 동일하고 문법만 다르다. TS 탭은 NestJS 기준인데 — React/바닐라 TS엔 DI 컨테이너가 없지만, NestJS가 Angular/Spring에서 영감받아 만들어져 Spring 개념이 거의 1:1로 대응된다(그래서 세션님한테는 오히려 지름길).

Kotlin + Spring 함정 Kotlin 클래스는 기본이 final이다. 그런데 Spring은 @Configuration·@Transactional·JPA 엔티티 등에서 프록시(상속)를 만들어야 해서, final이면 동작이 막힌다. 그래서 Spring Boot Kotlin 프로젝트는 kotlin-spring(all-open) 컴파일러 플러그인으로 Spring 어노테이션이 붙은 클래스를 자동으로 open 처리한다(start.spring.io로 만들면 기본 포함, JPA 엔티티는 kotlin-jpa 플러그인이 no-arg 생성자까지 해결). 커리큘럼 4단계의 "allopen, final 클래스" pitfall이 바로 이것.
04

Actuator

Actuator = 돌아가고 있는 백엔드 앱의 "상태를 들여다보고 관리하는" 창구를 자동으로 만들어주는 Spring Boot 기능이다.

스타터 의존성 하나 넣으면, 앱이 자기 상태를 알려주는 HTTP 엔드포인트들을 자동 생성한다.

엔드포인트하는 일
/actuator/health앱이 살아있는지, DB·외부 연동이 정상인지 (가장 많이 씀)
/actuator/metricsCPU·메모리(JVM 힙)·요청 수·응답시간 같은 지표
/actuator/info앱 버전·빌드 정보
/actuator/loggers실행 중에 로그 레벨을 실시간 변경
/actuator/env현재 적용된 설정값 확인
GET /actuator/health
{ "status": "UP", "components": {
    "db": { "status": "UP" },
    "redis": { "status": "UP" }
} }

왜 쓰나 — 프론트 접점

핵심은 운영·배포 환경에서 앱이 건강한지 자동 감시하기 위해서다. 프론트에도 은근히 닿는다:

· 로드밸런서·쿠버네티스가 /health를 주기적으로 찔러본다. UP이 아니면 트래픽을 끊거나 재시작시킨다. 배포된 서버가 잠깐 500을 뿜다 알아서 복구되는 뒤편엔 이 헬스체크가 있다.
· 모니터링 도구(Datadog·Prometheus)가 /metrics를 긁어간다. 백엔드 지표가 대시보드에 뜨는 경로가 바로 Actuator다.
· 장애 원인 판별 — API가 느릴 때 프론트 탓인지 백 탓인지, 백엔드는 이 메트릭(메모리 부족? 응답 지연?)으로 가른다.

프론트로 억지 비유하면, Next.js에 손수 만드는 /api/health 라우트나 웹앱 성능 지표를 자동 수집하는 RUM/헬스체크를 프레임워크가 공짜로 깔아주는 것에 가깝다.

보안 주의 /actuator/env·/actuator/heapdump는 설정값·내부 정보가 노출돼 민감하다. 실무에선 health 정도만 외부에 열고 나머지는 내부망에서만 접근하도록 잠근다.
05

번외 — SDD (Spec-Driven Development)

이날 곁다리로 나온 방법론. SDD = 명세 주도 개발 — "코드를 먼저 짜지 말고 명세(spec)를 먼저 작성해 그것이 진실의 원천이 되게 하자"는 접근이다. AI 코딩 에이전트가 등장하며 다시 주목받았다.

  1. 명세 작성 — 무엇을 만들지(요구사항·동작·제약)를 구조화된 문서로.
  2. 계획 수립 — 명세를 기술 설계·태스크로 분해.
  3. 구현 — 에이전트나 개발자가 그 계획대로 코드 생성.
  4. 검증 — 결과물이 명세를 만족하는지 확인.

핵심 아이디어는 "코드가 아니라 의도(intent)가 자산"이라는 것. 명세가 정확하면 구현은 재생성 가능하다고 본다. GitHub Spec Kit, AWS Kiro 등이 이 계열.

TDD와의 층위 TDD가 "테스트를 먼저"라면, SDD는 한 단계 위에서 "명세를 먼저"다. 참고로 Schema-Driven Development(스키마·OpenAPI·Protobuf를 먼저 정의하고 코드·타입을 생성)라는 동명이의어도 있다 — API/백엔드 문맥에선 이쪽을 뜻하기도 한다.
